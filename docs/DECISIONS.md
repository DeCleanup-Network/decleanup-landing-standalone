### Fifth-round — mobile horizontal-clip fix (~360–414px)
- **Symptom** — phone screenshots showed tech-stack, DMRV schematic, and DMRV intro (heading + audience tiles) clipped on the right. Root mechanism: `html,body{overflow-x:clip}` (anti rubber-band guard) **clips** anything wider than the viewport instead of scrolling, so the page never scrolls sideways — content just gets cut. Confirmed via `[...$$('*')].filter(e=>e.getBoundingClientRect().right>innerWidth)` at 390px → 50 clipped elements while `scrollWidth===clientWidth`.
- **Real cause** — grid items default `min-width:auto` (=min-content), so a wide logo / long heading inflates a `1fr` track past the viewport. `.builton-cell` ballooned to 232px (logo `max-width:80%` is a % → does NOT constrain min-content); the inline `repeat(3,1fr)` audience tiles hit ~410px min-content and, because `.dmrv-grid` stacks to one `1fr` column ≤1024px, that inflation cascaded into the schematic card (clipping it too).
- **Fixes** (`stylesheets/styles.css` + 2 JSX): added `img{max-width:100%}` (NO `height:auto` — it overrode the `height=20/24` attrs on the brand wordmark logos and ballooned them to natural 640px); `min-width:0` on `.builton-cell`, `.backers-grid>a`, `.dmrv-grid>*`; converted the two inline `repeat(3,1fr)` grids (audience tiles `sections.jsx`, hero KPIs `hero.jsx`) to `repeat(auto-fit, minmax(min(100%,150px),1fr))` (3-up desktop unchanged, ~2-up phone); lowered heading clamp minimums (`40px→28px` on the DMRV/built-on/steps h2/h3, hero h1 `52px→40px`) so long words fit ~360px.
- **Verified** — local preview (`.claude/preview-server.py`, dcu-root) at 360/390/414px: zero clipped elements except the glossary `.term-tip` (a hidden `opacity:0` tooltip, +11px, no page-scroll impact — left as known-minor). Desktop 1280px unchanged: stack 4-col, dmrv/hero 2-col, tiles/KPIs 3-col, `scrollWidth===clientWidth`. Screenshots captured for all three sections.
- **Gotcha for next time** — the preview harness **locks programmatic scroll** (`scrollTo`/`scrollTop` no-op); to screenshot a lower section, translate `#root` with `transform: translateY(-Ypx)`.
- **Glossary terms on mobile (`Term` in `primitives.jsx`)** — tapping a `?` term jumped to an unrelated part of the page instead of showing the tooltip. Cause: `.term` is a `tabIndex=0` span; on tap the browser focuses it and scrolls the focused element into view — for terms near the top under the fixed nav that's a large downward jump. (Desktop works because the tip shows on hover, no tap/focus.) Fix: `onMouseDown={e=>e.preventDefault()}` on the term — prevents focus-steal/scroll on tap while `onClick` still toggles `data-open` and keyboard Tab focus still works. Also restyled `.term-q` (12px→15px, weight 700, raised `top:-0.28em` so the badge sits off the dotted underline instead of looking crooked) and narrowed the mobile tooltip to `min(220px, calc(100vw-32px))` so a mid-line term's `left:0` tip stays within the viewport. Verified: all 5 terms open in place, no focus-steal, tips within viewport, 0 page overflow at 390px.
- **Glossary follow-up — the above didn't hold on real iOS.** User reported (even in incognito) that tapping a `?` still led nowhere / showed no explanation. Two likely iOS causes: (a) `onMouseDown preventDefault` can cancel the synthesized `click` on iOS → tooltip never opened; (b) the inline `position:absolute` tip + `:hover` sticky-hover behave unreliably on touch. **Reworked the interaction to be touch-bulletproof:** removed `onMouseDown`; tap toggles via `onClick`/`data-open` (+ Enter/Space, `role=button`, `aria-expanded`); close-outside now also listens to `touchstart`; `:hover` trigger gated behind `@media (hover: hover)` so iOS sticky-hover can't fire it; and on `≤640px` the tip becomes a **viewport-pinned fixed bottom sheet** (`position:fixed; left/right:16px; bottom:16px; z-index:60`, arrow hidden) — guaranteed visible on tap regardless of the term's scroll position or any focus-scroll. Confirmed safe because no term has a `transform`/`filter` ancestor that would re-anchor `position:fixed`. Desktop keeps the absolute hover/click tip near the term. Verified in preview (Chromium mobile emulation) at 390px; **real-iOS confirmation pending user re-test.**
- **The actual reason the user saw NO change after two deploys: browser caching.** `index.html` referenced `styles.css` / `*.jsx` as bare URLs, and `vercel.json` served them `Cache-Control: public, max-age=3600, must-revalidate`. Once a phone loaded the page, it reused the cached CSS/JSX for up to an hour **without revalidating** (normal reloads + same-session incognito included) — so neither the restyled `?` nor the new tap logic ever reached the device. Fix: (1) added a `?v=20260601c` cache-busting query to the `styles.css` + 6 `*.jsx` references in `index.html` (new URL ⇒ guaranteed cache miss ⇒ fresh fetch now; `index.html` itself is `max-age=0,must-revalidate` so the new references are seen on the next load); (2) switched the jsx + styles.css `Cache-Control` to `no-cache` in `vercel.json` so future deploys are picked up immediately via ETag revalidation. Also updated `.claude/preview-server.py` to strip `?query` before matching rewrites so local preview still serves the versioned URLs. **Lesson: for this CDN-Babel site, never ship asset changes without either bumping the `?v=` token or relying on the now-`no-cache` headers — bare-URL + long max-age silently hides every edit.**
- **Glossary rebuilt as a modal (the inline-tooltip approach was abandoned).** Even after cache-busting, the inline `position:absolute`/fixed tooltip kept failing on mobile (jump on tap, no visible explanation). Root frustration: it depended on focus/positioning that behaves unreliably on touch. **Final design: clicking a term opens its definition in a centered modal** built on the same `.modal-back`/`.modal` pattern as the working `StartModal`, **rendered via `ReactDOM.createPortal(..., document.body)`** so it's immune to scroll position, focus-scroll, and transformed/overflow ancestors. The term is `role=button` (click + Enter/Space), Escape/backdrop/× all close. The inline `.term-tip` is now a **desktop-hover-only** preview (`@media (hover: hover)`), and is `display:none` on touch (`@media (hover: none)`) so it can't sit off-screen. `data-open`, the click-outside `mousedown`/`touchstart` listeners, and the `onMouseDown preventDefault` were all removed. Verified open/close/center via synthetic clicks + screenshot at 390px (preview is `hover:hover`, so the touch `display:none` path is by-rule, not observed). **Tooling caveat learned: the MCP preview harness locks programmatic scroll and `preview_click` did not dispatch DOM-visible clicks — synthetic `el.click()` is what exercises the real React handler here; never claim a tap-interaction "works" from `withinVP`/`scrollY` checks alone.**
- **ACTUAL ROOT CAUSE (found after the above were all red herrings): the closed mobile nav panel was eating the clicks.** `.nav-mobile-panel` is `display:none` on desktop but `display:block` at ≤860px; when closed it's `opacity:0` + `transform` for the slide animation, but it had **no `pointer-events` override**, so it stayed hit-testable as a `position:fixed` layer covering the top ~617px of the viewport (`top:53px`, tall content). `document.elementFromPoint` at the hero eyebrow returned `a.nav-mobile-link[href="#what"]` — the *invisible* menu link, not the glossary term beneath it. So tapping a `?` actually clicked a hidden menu anchor → smooth-scroll to `#what` (~1514px = "jump to the middle of the page"), and the term's click never fired. This is why **every** earlier glossary fix was invisible (the tap never reached the term) and why desktop was fine (panel is `display:none` there). **Fix (one rule): `.nav-mobile-panel { pointer-events: none }` + `.nav-mobile-panel.open { pointer-events: auto }`.** Reproduced and verified with a REAL `preview_click` on a term that was in-viewport after a fresh load (scrollY stayed 0, no jump, modal opened) — the synthetic `el.click()` had masked it because it doesn't focus/hit-test through overlapping layers. **General lesson: when a tap "does nothing / jumps," check `document.elementFromPoint(x,y)` at the target — an invisible `opacity:0`/`pointer-events:auto` fixed overlay (closed menu/drawer) is a classic click thief; always pair visual-hide with `pointer-events:none`.**
